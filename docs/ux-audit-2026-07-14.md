# Auditoria de UX — 14/07/2026

Levantamento feito navegando pela aplicação real (dev server local apontando pro
Supabase de produção), pelos dois caminhos: **cliente novo comprando e
configurando do zero** e **admin interno da Alizo**. Org de teste usada:
`UX AUDIT Padaria Estrela (pode apagar)` + usuários `ux-audit-cliente@teste.alizo.local`
(admin da org) e `ux-audit-superadmin@teste.alizo.local` (super admin).

## Jornada da empresa cliente (self-service) — problemas encontrados

### P1 · Checkout é 100% simulado e termina num beco sem saída (CRÍTICO)
`/checkout` finge o pagamento com `setTimeout(2s)` e:
- **não cria conta, não cria org, não cria unidade** — nada vai pro banco;
- mostra "Pagamento confirmado! Enviamos as credenciais para o e-mail" — **mentira**, nenhum e-mail é enviado;
- o botão "Configurar meu funcionário IA agora" aponta pra `/dashboard/onboarding`,
  que redireciona pro `/login` — a pessoa que acabou de "pagar" cai numa tela
  pedindo uma senha que nunca existiu.
O motor de vendas do produto simplesmente não funciona ponta a ponta.

### P2 · Marca dividida: site/checkout são "AI Workforce OS" verde, produto é "alizo" ciano
Landing e checkout usam logo robô verde "AI Workforce OS"; login e dashboard usam
"alizo" ciano/índigo. Pro cliente parece ter comprado num site e caído em outro
produto. O hero da landing ainda colide com o navbar (título cortado por baixo do menu).

### P3 · Planos divergem entre site e checkout
Site oferece "Básico" (`/checkout?plan=basico`); o checkout só conhece
`starter|pro|enterprise` e silenciosamente vira "Plano Starter" — nome que o
cliente nunca viu. Existe ainda uma tabela `plans` no banco que o checkout ignora.

### P4 · Onboarding é decorativo (CRÍTICO)
O wizard de 5 passos em `/dashboard/onboarding`:
- não persiste progresso (estado React local — F5 zera tudo);
- passo "Conecte seu WhatsApp" é só texto mandando ir em outra tela (nenhum QR embutido);
- passo "Configure seu funcionário" tem inputs de nome/tom/script que **não salvam nada**;
- passo "Teste ao vivo" mostra uma conversa **hardcoded** e afirma "✓ Funcionou!
  Seu agente está respondendo corretamente" sem chamar agente nenhum;
- passo final proclama "Seu funcionário IA está ativo! 🎉" com nada ativo;
- links de "Documentação completa" e "Vídeo tutorial" apontam pra `#` (mortos).
Primeiro login pós-compra cai no **dashboard cheio de zeros**, não no onboarding.

### P5 · Tela da unidade pede infraestrutura interna pro cliente leigo (CRÍTICO)
`/dashboard/units/[id]` pede **URL da instância, nome da instância e API Key da
Evolution API**. Dono de padaria não tem como saber o que é isso. O backend já
suporta fallback pro servidor central (`EVOLUTION_API_URL`/`KEY` de env) e gera
nome de instância automático — os campos só deveriam aparecer pra equipe Alizo.

### P6 · Configurar o agente exige caçada de 3 níveis
Persona/horários/script ficam em Unidades → unidade → "Configurar agente".
A página "Agentes IA" do menu é quase toda estática (cards de marketing) e
considera "agente ativo" = unidade tem `whatsapp_phone`.

### P7 · Não existe teste de conversa real
Em nenhum lugar o cliente consegue conversar com o **seu** funcionário antes de
ligar pra valer (o `/chat` existente é o Kai de suporte da Alizo, não o agente
do cliente).

### P8 · Novos funcionários digitais falam "developerês" com o cliente
- Recrutador IA (vazio): "Abra uma vaga … ou via **POST /api/jobs**".
- Tráfego Pago (vazio): "Conecte contas via **POST /api/traffic/accounts** …
  passo a passo em **docs/setup/traffic-apis-setup.md**". KPIs "ROAS"/"CPA" sem explicação.
- Não há botão/formulário de "abrir vaga" na UI.

### P9 · Bugs visuais do tema escuro
- Card "Financeiro" do dashboard: valores em `text-green-800`/`text-red-800`
  (texto escuro sobre card escuro — ilegível).
- Card "Custo sistema" (visão super admin): fundo claro `rgba(248,250,252,…)`
  vazado no tema escuro.
- Saudação "Seu Workforce De IA Está Trabalhando — Julho De 2026" com
  `capitalize` aplicado na frase inteira, além de afirmar que algo "está
  trabalhando" quando nada foi configurado.

### P10 · Sidebar sobrecarregada e com linguagem técnica
13 itens pra um cliente recém-chegado ("CRM Pipeline", "Leads (lista)",
"Cobranças"…). "Funcionários" é o cadastro de funcionários **humanos** — colide
de frente com o conceito de "funcionário digital" que o site vende. "Onboarding"
é o último item do menu.

### P11 · Cliente não consegue trocar a própria senha
Settings → Segurança = "Em breve". O fluxo de provisionamento entrega senha
temporária — e não há como trocá-la na plataforma.

## Painel Admin Alizo — problemas encontrados

### P12 · Não existe painel admin de verdade
Super admin vê o mesmo dashboard do cliente com números de **todas as orgs
somados** (leads de todos os clientes num KPI só, "Boa tarde! Seu workforce está
trabalhando"). Não responde às perguntas da operação: quem travou no onboarding?
quem está sem WhatsApp? qual integração quebrou pra qual cliente?

### P13 · Página "Empresas" não diagnostica nada
Cards com contagens (unidades/WhatsApp/pessoas) e dois links — que passam
`?org=` pra páginas que **ignoram o parâmetro** (Unidades/Funcionários listam
tudo sempre). Não existe página de detalhe da empresa; pra investigar um cliente
é preciso abrir o banco.

### P14 · Gestão de acesso incompleta
`/dashboard/organizations/new` provisiona org+unidades+usuário com senha
temporária (bom!), mas depois disso não há: reset de senha, reativar/desativar
usuário, ver quem tem acesso.

## O que já funciona bem (aproveitar)

- Design system do dashboard (`components/ui/dashboard-ui.tsx`) — dark/glass/gradiente ciano-índigo consistente.
- Backend de conexão WhatsApp (connect/status com QR + polling) e fallback de
  Evolution central por env.
- Provisionamento via `/api/admin/users` (service role) com senha temporária.
- RLS multi-tenant sólido (org só vê o que é dela; super_admin vê tudo).
- Motores reais: conversation-engine (SDR), recruiter (validado ponta a ponta), traffic.
- Webhook de intake de leads com UI razoável em Settings.

## Plano de correção (implementado nesta sessão)

1. **Checkout real**: criar org + unidade + usuário + sessão de verdade
   (`/api/checkout/complete`, service role), com senha escolhida pelo cliente e
   login automático → onboarding. Pagamento continua marcado como pendente de
   integração (trial transparente), sem mentir sobre e-mail.
2. **Rebrand do site/checkout** pra identidade Alizo e planos unificados com o produto.
3. **Onboarding data-driven**: progresso derivado do banco (conta → WhatsApp →
   funcionário → teste → ativo), QR code embutido, config salva de verdade,
   teste de conversa usando o motor real em modo sandbox, ativação real.
4. **Dashboard do cliente**: card "o que fazer agora", status dos funcionários
   digitais contratados, correção dos bugs de tema, linguagem leiga.
5. **Sidebar por papel**: cliente vê menu enxuto em linguagem simples; itens
   técnicos/internos só pra super admin.
6. **Unidade**: campos Evolution visíveis só pra super admin; conexão WhatsApp
   com QR inline pro cliente.
7. **Recruiter/Tráfego**: empty states leigos + formulário "Nova vaga" na UI.
8. **Troca de senha** funcional em Settings.
9. **Admin**: página Empresas com saúde/onboarding por cliente + página de
   detalhe com diagnóstico e ações (reset de senha, ativar/desativar).
